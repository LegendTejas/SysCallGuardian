# 🚀 SysCallGuardian — Git & GitHub Sync Guide

This guide explains how to initialize a local project, link it to a GitHub repository, and maintain a seamless synchronization workflow.

---

## 1. Initial Setup: Linking Local to Remote

If you have a local folder (like **SysCallGuardian**) and want to push it to a **new** GitHub repository for the first time:

### Step A: Initialize Git

Open your terminal in the project root and run:

```bash
git init
```

### Step B: Link the Remote Repository

Replace the URL with your actual GitHub repository URL:

```bash
git remote add origin https://github.com/LegendTejas/SysCallGuardian.git
```

---

## 2. The "Daily Workflow" (Syncing Changes)

Whenever you make changes to your code, follow these **three steps** to push them to GitHub:

| Step                | Command                               | Description                                |
| :------------------ | :------------------------------------ | :----------------------------------------- |
| **1. Stage**  | `git add .`                         | Prepares all changed files for a commit.   |
| **2. Commit** | `git commit -m "brief description"` | Saves your changes locally with a message. |
| **3. Push**   | `git push origin main`              | Uploads your local commits to GitHub.      |

> [!TIP]
> Use descriptive commit messages (e.g., `git commit -m "Added validation logic for syscalls"`) to keep track of your progress.

---

## 3. Handling Conflicts & Remote Updates

If you (or a teammate) make changes directly on GitHub, your local version will be "behind." You must pull those changes before you can push again.

### Standard Pull

```bash
git pull origin main
```

### If you get "Unrelated Histories" error:

If you initialized a repo locally and it already exists on GitHub with different files, use:

```bash
git pull origin main --allow-unrelated-histories
```

---

## 4. Best Practices: Using `.gitignore`

Always ensure you have a `.gitignore` file in your root directory. This prevents "junk" files (like virtual environments or local secrets) from being uploaded.

**Example `.gitignore` for Python:**

```text
venv/
__pycache__/
.env
*.db
.vscode/
```

> [!IMPORTANT]
> **Never commit your `.env` file.** It often contains sensitive API keys or passwords. Always add it to your `.gitignore`.
